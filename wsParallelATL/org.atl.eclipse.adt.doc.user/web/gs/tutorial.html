    Creating ATL Files and ATL Projects

    It is essential to know how to create an ATL Project and an ATL file.

    To create an ATL project select the “File” menu, choose “New”, then “Project” and finally “ATL / ATL Project”. Enter a project name and press the “Finish” button.

    An ATL file is created in the ATL project folder by selecting “New” in the menu “File”, then choosing “File”. Enter the file name and do not forget the extension “.atl”. Press “Finish”.

    From Classes to Relational Tables

    In the following, it will be explained how to implement the transformation from classes to relational tables step by step.

    In general, classes are mapped to tables and class attributes to table attributes. However, if attributes are not single valued or not of a primitive type (Datatype), this is different. Attributes that are of the type Class must be transformed to foreign keys. For multivalued attributes a separate table (the values table) has to be created. This values table must have two columns, one containing the foreign key and another one containing the values. The idea is that if an attribute has five values, the values table contains five rows. The transformation does not concern the data that will be put in the tables.

    Simplifications of this class to relational description:

        - It does not take inheritance into account.

        - It does not take the classifier isAbstract into account.

        - Naming collisions are not cought. The class model must not contain attribute names such as objectId.

        - In the class model there must be at least one attribute of type 'Integer'

    Metamodels

    Before an ATL program can be written you must have the target and the source metamodels. In case of the class to relational transformation, the Class metamodel is the source metamodel and the Relational metamodel is the target metamodel.

    The Class metamodel (see Figure 1 Class) consists of classes having a name which they inherit from the abstract class NamedElt. The principal class is the class Class, which contains a set of attributes of the type Attribute and has the super references pointing to superclasses for modelling inheritance trees. The class DataType models primitive data types. Class and DataType inherit from Classifier which serves to declare the type of Attributes. Attributes can be multivalued, which has an important impact on the transformation.


    Figure 1: Class Metamodel

    The Relational metamodel (see Figure 2) consists of classes having a name which they inherit from the abstract class Named. The principal class Table contains a set of Columns and has a reference to its keys. The class Column has the references owner and keyOf pointing to the Table it belongs to and of which it is part of the key (in case it is a key). Furthermore, Column has a reference to Type.



    Figure 2: Relational Metamodel

    The Header and the Import Section

    Each ATL module starts with the declaration of the module name and the source and target (meta-) models. In this case, the module Class2Relational has the source (meta-)model Class and the target (meta-)model Relations. The import section is optional. The Class2Relational module requires the library strings which is declared with the uses expression.

    module Class2Relational;

    createOUT : Relational fromIN : Class;

    usesstrings;

    The Helpers

    Most ATL programs have helper variables or helper functions which serve mostly to implement repretitive pieces of code in one place. The Class2Relational has only one helper.

    The Global Helper Variable objectIdType

    ObjectIdType description:

    In the Class2Relational  module, the type for primary and foreign keys has to be defined  as Integer by the means of a global helper variable, called objectIdType.

    ObjectIdType implementation:

    helper def: objectIdType : Relational!Type =

          Class!DataType.allInstances()->select(e | e.name = 'Integer')->first();

    This helper variable is globally defined which means that the variable can be reused by the rules of this module.
    The OCL expression of this helper takes all instances of the source model with type DataType and selects the type with the name “Integer”. The selection returns a sequence with (we assume here) one DataType instance named 'Integer'. The OCL operation first returns the first instance, meaning the DataType instance named 'Integer'.

    Please note: In ATL, it is possible to navigate in the source model, but not in the target model. References to source model elements are automatically translated into corresponding references of the target model.

    The Rules

    In this chapter the helpers and rules of the Class2Relational transformation will be explained one by one.

    The DataType2Type Rule (a simple rule)

    DataType2Type rule description:

    For each DataType instance, a Type instance has to be created.
        -  Their names have to correspond.

    DataType2Type rule implementation:

    rule DataType2Type {

          from

                dt : Class!DataType

          to

                out : Relational!Type (

                      name <- dt.name

                )

    }

    The DataType2Type rule creates for each DataType a new Type instance and copies the name attribute.
    The rule begins with the keyword rule, the rule's name and an opening bracket.
    The from-part declares that this rule will be called (mached) for each DataType instance of the source model.
    The to-part instanciates a new Type instance of the target model for each mached instance.
    The left arrow operator assigns the name of the matched DataType to the name of  the new type instance.
    The rule finishes with a closing bracket.

    The Class2Table Rule (a rule with two target pattern elements)

    Class2Table rule description:

    For each Class instance, a Table instance has to be created.
        -    Their names have to correspond.
        -    The col reference set has to contain all Columns that have been created for single-valued attributes and also the key described in the following.
        -    The key reference set has to contain a pointer to the key described in the following.
        -    An Attribute instance has to be created as key.
              Its name has to be set to ‘objectId’, its type to String.

    Class2Table rule implementation:

    rule Class2Table {

          from

                c : Class!Class

          to

                out : Relational!Table (

                      name <- c.name,

    -- Columns are generated from Attributes in another rule not 

    -- explicitly called here !

                      col <- Sequence {key}->

    union(c.attr->select(e | not e.multiValued)),

                      key <- Set {key}

                ),

                key : Relational!Column (

                      name <- 'objectId',

                      type <- thisModule.objectIdType

                )

    }

    For each matched class (there is no filter so all classes are matched), the Class2Table rule creates a table (target pattern element with variable out) and a key (target pattern element with variable key).
    The values of attributes are set with the “<-” operation.The col attribute of  table is a sequence. This sequence contains the key column plus what is addeded by the the union command. The union command adds all those columns that are created for single valued attributes of the matched class. The  multi valued attributes are filtered bu the select operation. The key attribute is a sequence that consists of one element, namely the key created by the target pattern element with variable key. For the creation of the key type, the helper function objectIdType is called and the name of the key column is set to "objectID".

    The SingleValuedDataTypeAttribute2Column Rule (a rule with filter)

    SingleValuedDataTypeAttribute2Column  rule description:

    For each single-valued Attribute instance of the type DataType, a Column instance has to be created
        - Their names and their types have to correspond.

    SingleValuedDataTypeAttribute2Column  rule implementation:

    rule SingleValuedDataTypeAttribute2Column {

          from

                a : Class!Attribute (

                      a.type.oclIsKindOf(Class!DataType) and not a.multiValued

                )

          to

                out : Relational!Column (

                      name <- a.name,

                      type <- a.type

                )

    }

    The from-part of a rule defines for which elements of the source model the rule will be executed. There is always a restriction through the source type of the variable that is specified (in this case the rule matches only for Class!Attribute instances) . However, in the from-part it is also possible to specify a filter that further constraints the execution of the rule. The filter is added in brackets at the end of the from-part.
    The SingleValuedDataTypeAttribute2Column rule has a filter in the from-part that checks that the attribute's type is (oclIsKindOf) a DataType and is singlevalued.
    The name attribute and the type reference are copied. Again, the reference to the source model is automatically translated to the corresponding reference of target model  (type <- a.type).

    The MultiValuedDataTypeAttribute2Column Rule

    The MultiValuedDataTypeAttribute2Column rule decription:

    For each multi-valued Attribute instance of the type DataType, a Table instance has to be created.
        - The Table’s name is the name of the Attribute’s Class concatenated with an underscore and the name of the Attribute.
            - The col reference set has to reference the two Columns described in the following.
                - An identifier Column instance has to be created.
                    - Its name has to be set to the Attribute’s class name concatenated with ‘Id’
                    - Its type reference has to reference a Type with the name Integer
                - A Column instance has to be created to contain the values of the Attribute.
                    - The names and their types have to correspond.

    The MultiValuedDataTypeAttribute2Column rule implementation:

    rule MultiValuedDataTypeAttribute2Column {

          from

                a : Class!Attribute (

                      a.type.oclIsKindOf(Class!DataType) and a.multiValued

                )

          to

                out : Relational!Table (

                      name <- a.owner.name + '_' + a.name,

                      col <- Sequence {id, value}

                ),

                id : Relational!Column (

                      name <- a.owner.name.firstToLower() + 'Id',

                      type <- thisModule.objectIdType

                ),

                value : Relational!Column (

                      name <- a.name,

                      type <- a.type

                )

    }

    The MultiValuedDataTypeAttribute2Column rule creates a table, an id column and a value column. Its filter declares that it matches only attributes which are DataTypes and singlevalued.

    The ClassAttribute2Column Rule

    The ClassAttribute2Column rule decription:

    For each single-valued Attribute of the type Class, a new Column has to be created.
        - Its name has to be set to the attribute’s name concatenated with ‘id’.
        - Its type reference has to reference a Type with the name Integer

    The ClassAttribute2Column rule implementation:

    rule ClassAttribute2Column {

          from

                a : Class!Attribute (

                      a.type.oclIsKindOf(Class!Class) and not a.multiValued

                )

          to

                out : Relational!Column (

                      name <- a.name + 'Id',

                      type <- thisModule.objectIdType

                )

    }

    The ClassAttribute2Column rule creates a Column with the attribute’s name concatenated with ‘id’.

    The MultiValuedClassAttribute2Column Rule

    The MultiValuedClassAttribute rule decription:

    For each multi-valued Attribute of the type Class, a new Table has to be created.
        - The Table’s name is the name of the Attribute’s Class concatenated with an underscore and the name of the Attribute.
            - The col reference set has to reference the two Columns described in the following.
            - An identifier Column instance has to be created.
                - Its name has to be set to the Attribute’s class name concatenated with ‘Id’.
                - Its type reference has to reference a Type with the name Integer which - if not yet existing - has to be created.
            - A foreign key Column instance has to be created.
                - Its name has to be set to the Attribute’s name concatenated with ‘Id’.
                - Its type reference has to reference a Type with the name Integer

    The MultiValuedClassAttribute rule implementation:

    rule MultiValuedClassAttribute2Column {

          from

                a : Class!Attribute (

                      a.type.oclIsKindOf(Class!Class) and a.multiValued

                )

          to

                out : Relational!Table (

                      name <- a.owner.name + '_' + a.name,

                      col <- Sequence {id, foreignKey}

                ),

                id : Relational!Column (

                      name <- a.owner.name.firstToLower() + 'Id',

                      type <- thisModule.objectIdType

                ),

                foreignKey : Relational!Column (

                      name <- a.name + 'Id',

                      type <- thisModule.objectIdType

                )

    }

    The MultiValuedClassAtrribute2Column creates for each multivalued attribute that is not a primitive type, a table holding two columns.

    Typical Test Example

    The test example that will be shown in the following has been implemented using KM3. It concerns the package sample with the classes Family and the class Person that will be transformed to the corresponding relational model.

    The model of the classes Family and Person:

    package Sample {

     

          datatype String;

          datatype Integer;

     

          class Family {

                attribute name : String;

                attribute members[*] : Person;

          }

     

          class Person {

                attribute firstName : String;

                attribute closestFriend : Person;

                attribute emailAddresses[*] : String;

          }

    }

    is transformed to the following model of tables and columns:

    table Person {

            primary column objectId : Integer;

            column firstName : String;

            column closestFriendId : Integer;

    }

     

    table Family_members {

            column familyId : Integer;

            column membersId : Integer;

    }

     

    table Person_emailAddresses {

            column personId : Integer;

            column emailAddresses : String;

    }

     

    table Family {

            primary column objectId : Integer;

            column name : String;

    }

    Execution

    In Eclipse, there is a specific launch configuration for ATL programs.
    After a right button mouse click on the atl file in the Eclipse navigator choose "Run" or "Debug".
    Create a new configuration by clicking on the appropriate button.
    Enter the program name that you wish to execute and the models that it requires.

    Please note:
        - The configuration is case sensitive
        - The configguration and your file must correspond (e.g. the metamodel names used must be identical)
